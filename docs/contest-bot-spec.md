## 2. プロジェクト概要

Discord 上のコマンドから、AtCoder Problems のバーチャルコンテスト（以下、バチャ）を自動生成する個人用 Bot を作成する。

主な利用フローは以下。

1. Discord で `/start` または `/custom-start` を実行する
2. Bot が AtCoder Problems の問題情報と自分の提出情報をもとに問題集合を決定する
3. Bot が AtCoder Problems の internal API を用いてバチャを作成する
4. Discord にバチャ URL と開始時刻を返す

本システムは **個人用** であり、一般公開サービスとしての設計は行わない予定。

---

## 3. 目的

本プロジェクトの目的は以下。

- AtCoder Problems 上でのバチャ作成を自動化する
- 未 AC 問題を優先したバチャを簡単に作れるようにする
- 難易度帯・問題数・コンテスト時間・出典を柔軟に指定できるようにする
- 同条件でのバチャ乱立を防ぐ
- Cloudflare 無料枠で実用的に運用する

---

## 4. 非目的

以下は本プロジェクトの対象外とする。

- 不特定多数のユーザー向け公開サービス化
- 複数ユーザー対応
- Web UI の提供
- バチャ開始通知・終了通知
- 再抽選 UI
- プリセット機能
- `/recent` コマンド
- 自動結果集計
- AtCoder 本体との認証連携
- AtCoder Problems の internal API 変更への永続的追従性の保証

---

## 5. 想定利用者

- 利用者は 1 名のみ
- 利用者は Discord と AtCoder Problems を日常的に使う
- 利用者は AtCoder の user ID を固定で持つ
- 利用者は Bot を自分用として運用する

## 6. 重要前提

### 6.1 AtCoder Problems の internal API 依存

バチャ作成は AtCoder Problems の internal API に依存する。
これは問題ない前提とする。

internal API の仕様変更や破壊的変更が入った場合は、その時点で実装を修正する。

### 6.2 提出情報の正

提出情報の正は AtCoder Problems とする。
独自に別の正を持たない。

ただし、AtCoder Problems API を毎回フルに叩かないよう、同期結果のキャッシュは保持する。

### 6.3 開始時刻

開始時刻は、**コマンド実行時点から見て次の区切り時刻**とする。

例:
- 12:03 に 5 分区切りなら 12:05 開始
- 12:05 ちょうどなら 12:10 開始

「開始時刻になってからバチャを作る」のではなく、**未来の開始時刻を指定してその場でバチャを作る**方式を採用する。

### 6.4 通知

バチャ作成後の通知は不要。
Discord には作成直後に URL と開始時刻だけを返せばよい。

### 6.5 AtCoder Problems API のアクセス間隔

AtCoder Problems へのアクセスは、最低でも **5 秒以上** 間隔を空けること。
これは同期処理・バチャ作成処理の両方で守る。

## 7. 採用技術スタック

### 7.1 確定スタック

- 言語: TypeScript
- HTTP / ルーティング: Hono
- API スキーマ管理: OpenAPI
- API リファレンス表示: Scalar
- 実行基盤: Cloudflare Workers
- 排他制御 / 重複作成防止: Durable Objects
- 永続 DB: Cloudflare D1
- ORM: Drizzle ORM
- バリデーション: Zod または同等ライブラリ
- テスト: Vitest
- デプロイ / ローカル開発: Wrangler

### 7.2 スタック採用方針

- Cloudflare Workers を中心に完結する構成を採用する
- Durable Object は「状態の本体」ではなく「排他と重複作成防止の番人」として使う
- 長時間かかる同期処理は `waitUntil()` で引き延ばさず、Durable Object + Alarm で 1 バッチずつ進める
- 永続データの主保存先は D1 とする
- D1 へのアクセスには Drizzle ORM を使う
- API 仕様は OpenAPI で管理する
- API 仕様と実装の乖離を避けるため、OpenAPI はコードから生成する
- API リファレンス表示には Scalar を使う
- 実装は Cloudflare ネイティブとの相性を優先する

## 8. 外部サービスとの関係

### 8.1 Discord

Discord は slash command の入力口として使う。

必要なコマンドは以下のみ。

- `/start`
- `/custom-start`
- `/setting`
- `/init`

### 8.1.1 Discord command の登録方針

- 開発時は guild command を使って即時反映で検証する
- 公開配布前提ではないため、MVP では guild command 運用を優先する
- command 登録手順そのものは README に書く

### 8.1.2 Discord interaction endpoint

- Discord interaction は Cloudflare Workers 上の webhook 受け口として処理する
- Discord の署名検証を必須とする
- この endpoint は公開 API というより外部サービス連携用の入口なので、MVP では公開 OpenAPI には載せない

### 8.2 AtCoder Problems

AtCoder Problems から利用するものは以下。

- 問題情報
- 難易度情報
- コンテストと問題の対応情報
- 提出情報
- バチャ作成 internal API

### 8.3 Cloudflare

Cloudflare は以下の役割を持つ。

- Discord interaction endpoint の実行
- 排他制御
- データ永続化
- ログ保存

## 9. コマンド仕様

## 9.1 `/start`

### 概要
保存済みデフォルト設定を使ってバチャを作成する。

### 挙動
1. デフォルト設定を読み込む
2. 提出情報の増分同期を行う
3. 次の区切り時刻を計算する
4. 問題候補を絞り込む
5. バチャを作成する
6. URL と開始時刻を Discord に返す

### 返却内容
成功時は以下のみ返す。

- バチャ URL
- 開始時刻

### 補足
- 通常メッセージで返す
- ephemeral は使わない

---

## 9.2 `/custom-start`

### 概要
今回だけ条件を上書きしてバチャを作成する。

### 挙動
1. デフォルト設定を初期値として読む
2. 入力された上書き条件を反映する
3. 提出情報の増分同期を行う
4. 次の区切り時刻を計算する
5. 問題候補を絞り込む
6. バチャを作成する
7. URL と開始時刻を Discord に返す

### 返却内容
成功時は以下のみ返す。

- バチャ URL
- 開始時刻

---

## 9.3 `/setting`

### 概要
デフォルト設定の操作を行う。

### 役割
- デフォルト設定の表示
- デフォルト設定の更新

### 注意
通常運用時に使うコマンドではなく、設定変更時に使う。

---

## 9.4 `/init`

### 概要
初期同期関連の操作を行う。

### 役割
- 初回フル同期の実行
- 同期状態の確認

### 注意
通常運用時に使うコマンドではなく、初期設定時に使う。

---

## 10. 設定項目

`/setting` で変更可能な項目は少なくとも以下。

- AtCoder user ID
- デフォルトの開始時刻区切り単位
- デフォルトの問題数
- デフォルトのバチャ時間
- デフォルトの難易度帯指定
- 未 AC フィルタの ON/OFF
- 出典フィルタ
- penalty second
- experimental difficulty を含めるか
- 公開設定
- タイトルテンプレート
- メモテンプレート

### 10.1 開始時刻区切り単位

サポートする値は少なくとも以下。

- 5 分
- 10 分
- 15 分
- 30 分

既定値は **5 分** とする。

### 10.2 難易度指定

複数帯形式を正式採用する。

例:
- 800〜999 を 2 問
- 1000〜1199 を 2 問
- 1200〜1399 を 1 問

単一範囲形式は簡易入力として扱う。

### 10.3 出典フィルタ

MVP で以下を対象にする。

- ABC
- ARC
- AGC
- その他を含めるかどうか

### 10.4 D1 上の設定保存方針

設定は D1 の `settings` を本体として保存する。

- `settings` には単一値の設定を保持する
- 出典フィルタは少なくとも `ABC` `ARC` `AGC` `その他` を個別に表現できるようにする
- 難易度帯指定は `settings` に直接 JSON で埋め込まず、別レコードで 1 件ずつ保持する

### 10.5 難易度帯指定の保存形式

複数帯形式を扱うため、難易度帯指定は 1:N の構造で保持する。

少なくとも以下の情報を帯ごとに持てるようにする。

- 並び順
- 難易度下限
- 難易度上限
- その帯から何問取るか

## 11. 問題選定仕様

### 11.1 基本方針

問題は Bot 側で選定する。
AtCoder Problems のバチャ作成 API に「未 AC だけで作る」といった条件を渡してサーバー側に選ばせるのではなく、**最終的な問題配列をこちらで作って渡す**。

### 11.2 絞り込み条件

問題候補の絞り込みは少なくとも以下で行う。

- 出典フィルタ
- 難易度フィルタ
- 未 AC フィルタ
- 問題数
- experimental difficulty の扱い
- 直近使用問題除外

### 11.2.1 問題カタログの扱い

- `merged-problems.json` と `problem-models.json` と `contest-problem.json` をもとに問題カタログを構築する
- 問題カタログは D1 にキャッシュする
- 毎回外部取得せず、一定期間ごとに再同期する
- 問題選定は D1 にキャッシュした問題カタログを前提に行う

### 11.2.2 候補内での選び方

- 条件に合う候補が複数ある場合は、候補内からランダムに選ぶ
- ただし必要問題数を満たせない場合はエラーにする

### 11.3 直近使用問題除外

同じ問題が短期間で何度も出るのを避けるため、直近に使用した問題は一定期間除外できるようにする。
この期間はデフォルト設定として保持する。

### 11.4 候補不足

必要問題数を満たせない場合はエラーにする。
自動的に条件を緩和しない。

## 12. 提出同期仕様

## 12.1 基本方針

提出情報は AtCoder Problems を正とする。
ただし、毎回フル取得はしない。

D1 に以下を保持し、以後は増分同期を行う。

- 最終同期時刻
- AC 済み問題集合
- 必要な同期チェックポイント
- 問題カタログのキャッシュ

実装上は、少なくとも同期状態と AC 済み問題集合を D1 に保持できる構造にする。

## 12.2 初回フル同期

未 AC フィルタを正しく機能させるため、初回のみ過去提出をさかのぼってフル同期する。

### 実行方法
- `/init` から実行する
- 初回セットアップ時に 1 回だけ行う想定

### 進捗表示
- `/init action:status` で状態だけ確認できればよい
- 進捗の本体は D1 の同期状態に保存する

### 失敗時
- エラーを返す
- 次回は前回成功地点の少し前から再開する
- 重複は最終的に除去する

### 実装方針
- 冪等にする
- 再実行しても壊れないようにする
- 1 回の Worker invocation では 1 バッチだけ処理する
- バッチ継続は Durable Object の Alarm で再スケジュールする

## 12.3 通常同期

`/start` と `/custom-start` のたびに、毎回増分同期する。

### 理由
- 常に最新の未 AC 情報に近づけたい
- API 呼び出し回数を抑えつつ正しさを保ちたい

### 補足
- 毎回「フル同期」ではない
- 毎回「前回同期以降だけ」を同期する

## 13. バチャ作成仕様

## 13.1 作成手順

バチャ作成は以下の順で行う。

1. 問題候補を決定する
2. AtCoder Problems にコンテスト本体を作成する
3. そのコンテストに問題配列を流し込む
4. 作成結果を保存する
5. Discord に URL と開始時刻を返す

## 13.2 internal API 依存

本実装は AtCoder Problems の internal API を直接利用する。

AI エージェントは、バチャ作成処理の実装時に、以下の URL および対応ソースコード位置を前提にしてよい。

### 本番 URL
- `POST https://kenkoooo.com/atcoder/internal-api/contest/create`
- `POST https://kenkoooo.com/atcoder/internal-api/contest/item/update`

### 参考ソース
- `https://github.com/kenkoooo/AtCoderProblems/blob/master/atcoder-problems-backend/src/server/endpoint/internal_api/contest/mod.rs`
- `https://github.com/kenkoooo/AtCoderProblems/blob/master/atcoder-problems-backend/src/server/endpoint/internal_api/contest/item.rs`

### 補足
既存実装例として、以下も参考にしてよい。

## 14. 重複作成防止

### 14.1 目的

同じ条件・同じ開始時刻でバチャが乱立することを防ぐ。

### 14.2 実装責務

Durable Object が以下を担う。

- 同時実行の排他
- 同一条件・同一開始時刻の二重作成防止
- 作成途中の状態管理
- 既存作成済みバチャの返却
- 長時間同期ジョブの 1 バッチずつの再開制御

### 14.4 導入タイミング

Durable Object は contest 作成処理だけでなく、初回提出同期の長時間ジョブ制御にも使う。
状態本体は D1 に置き、Durable Object は実行制御と再開の番人として使う。

### 14.3 期待動作

- すでに作成済みのバチャがある場合、新規作成せず既存 URL を返す
- 途中で失敗した場合も、作成済みかどうかを確認する
- 「実は立っていた」ケースでは必ず既存 URL を返す

## 15. ログ仕様

ログは比較的しっかり保存する。

少なくとも以下を残す。

- コマンド実行時刻
- コマンド種別
- Discord の実行コンテキスト
- 使用設定の要約
- 同期開始・終了
- API 呼び出し結果
- 候補数
- 作成結果
- 作成済み URL
- 重複作成防止の判定結果
- エラー内容

### 15.1 保存先

- 主保存先は D1
- 必要に応じて Worker ログにも出す

実装上は、少なくとも command 実行ログと contest 作成履歴を D1 に保存できる構造にする。

### 15.2 目的

- デバッグ
- 失敗時の原因調査
- 重複作成防止の検証
- 同期不整合の確認

## 16. エラー処理方針

### 16.1 成功時
返す内容は以下のみ。

- バチャ URL
- 開始時刻

### 16.2 失敗時
返す内容は以下。

- エラー原因

### 16.3 例外
処理途中でエラーになっても、すでにバチャが作成済みであることが確認できた場合は、エラーではなくその URL を返す。

### 16.4 エラー時の再実行
ユーザーが再度コマンドを打ち直す前提でよい。
Bot 側は過度な自動リトライをしない。

## 17. セキュリティ方針

### 17.1 token の扱い

AtCoder Problems 用 token は Discord から入力させない。
Cloudflare の Secret に保存する。

### 17.2 ログ

以下をログに出してはいけない。

- token
- Cookie
- 認証ヘッダ
- 秘密値

### 17.3 ドキュメント分離

token 更新手順は README に直接書かず、別ドキュメントに切り出す。

### 17.4 secret と設定ファイルの分離

- token や秘密値は Cloudflare Secret に保存する
- `wrangler` 設定ファイルや DB 識別子は秘密情報としては扱わない
- ローカル開発時の secret 管理手順は README に書く

1. `/start` でデフォルト設定からバチャを作成できる
2. `/custom-start` で条件上書きして作成できる
3. `/setting` で設定変更と初回フル同期ができる

- 本プロジェクトは「個人用」であることが重要

- 通常時の快適さを重視する
